# Ternary-Zero Core: W2A16 GEMV Kernel Implementation Plan

## Project Overview
A high-performance 2-bit Weight / 16-bit Activation **GEMV** (General Matrix-Vector multiplication) kernel for RTX 4060, using ternary weights {-1, 0, 1} with PTX-level bit manipulation and a Rust orchestration layer.

### Scope: GEMV, Not GEMM

This kernel computes **y = W × x** where W is a 2D weight matrix and x is a 1D activation vector. This is explicitly a **GEMV** implementation, not GEMM (General Matrix-Matrix multiplication).

| Operation | Dimensions | This Kernel | Target Use Case |
|---|---|---|---|
| **GEMV** | `[M,N] × [N] → [M]` | **Optimized** | Autoregressive decoding (token-by-token), edge inference, low-batch serving |
| **GEMM** | `[M,N] × [N,K] → [M,K]` | **Not supported** | High-throughput datacenter batching (prefill, parallel decoding) |

**Why GEMV matters:** In autoregressive language model inference (e.g., LLM token generation), each decoding step is a GEMV — a single token's activations multiplied against each layer's weight matrix. This is the latency-critical path. GEMM (batched inference, prompt prefill) is a different optimization target requiring tiled matrix-matrix algorithms (e.g., CUTLASS, cuBLAS) that are outside this kernel's scope.

**Positioning:** This is a **specialized inference kernel for memory-constrained, latency-sensitive, low-batch scenarios** — not a general-purpose matrix multiplication library. For high-throughput serving with large batch sizes, the memory bandwidth advantage of ternary compression diminishes relative to the compute throughput of GEMM, and standard FP16/BF16 GEMM kernels (cuBLAS, FlashAttention) may be more appropriate.

## Phase 2: CUDA Kernel — `kernel/ternary_zero.cu`

### Step 2.1: 2-Bit Packing & Memory Layout

**Encoding Scheme:**
| Ternary Value | 2-bit Code | Notes |
|---|---|---|
| 0  | `00` | Zero-gate (skip) |
| +1 | `01` | Add activation |
| -1 | `10` | Subtract activation |
| 11 | (unused) | Invalid / sentinel |

**Packing:**
- 16 ternary weights → 1 `uint32_t` (32 bits / 2 bits each)
- Bit layout: `[w0:lsb | w1 | w2 | ... | w15:msb]`

**Column-Major Tiled Layout:**
```
// Weight matrix W[M, N] stored as packed uint32_t
// Tile: 64 rows × 1024 cols (packed into 64 × 64 uint32_t tiles)
// Each uint32_t holds 16 consecutive weights along N
// Memory access pattern: threads in a warp read consecutive uint32_t → coalesced 128-byte
```

### Step 2.2: PTX Inline Assembly — Bit-Level Decompression

**Key PTX Instructions:**
1. **`PRMT` (Permute Byte):** Rearrange bytes within a register to position 2-bit fields for extraction
2. **`LOP3.LUT` (3-input Logic):** Apply arbitrary 3-input boolean function via lookup table for masking/mixing bits
3. **BFI / SHL / SHR:** Bit-field insert and shifts for extraction

**Decompression Algorithm (16 weights from uint32_t):**
```
// Input: uint32_t packed (16 weights × 2 bits)
// Strategy: Process 4 weights per iteration using mask + shift
// Use PRMT to pre-swap bytes into convenient positions
// Use LOP3.LUT to simultaneously mask 2-bit fields and classify {00,01,10}

asm volatile(
    "prmt.b32 %0, %1, 0, 0x3210;"   // byte-permute for alignment
    : "=r"(permuted) : "r"(packed));

// For each 2-bit weight w[i]:
//   sign[i] = (w[i] >> 1) & 1    — bit 1: sign flag (1 = negative)
//   mag[i]  = w[i] & 1            — bit 0: magnitude (1 = non-zero)
//   skip[i] = (w[i] == 0)         — both bits zero

// LOP3.LUT truth table for (sign, mag, 0) → accumulate_decision:
//   00 → skip (zero weight)
//   01 → add  (positive)
//   10 → sub  (negative)
//   11 → invalid
```

**Register Budget:**
- ~8 regs for packed weights
- ~8 regs for unpacked sign/mag masks
- ~8 regs for activation staging
- ~8 regs for accumulators
- ~8 regs for temporaries/addresses
- Total: ~40 registers → within 32-64 target, enables 2+ blocks/SM

### Step 2.3: Zero-Gate Arithmetic Logic

```cuda
// Instead of: acc += (float)weight * activation;  (standard FMA)
// Use branchless integer conditional:

// Decode 2-bit weight into (sign_bit, nonzero_bit)
// sign = (weight_bits >> 1) & 1
// nz   = weight_bits & 1

// Conditional accumulation (branchless):
// acc = sign ? (acc - act) : (acc + act)   when nz=1
// acc = acc                                 when nz=0 (zero-gate, skip)

// Implementation with predicated execution:
//   int mask_nz = -nz;           // 0xFFFFFFFF if non-zero, 0x00000000 if zero
//   int mask_neg = -sign;        // 0xFFFFFFFF if negative
//   half2 delta = mask_neg ? -act : act;
//   acc += (delta & mask_nz);    // zero-gated add
```

### Step 2.4: Kernel Signature & Grid Configuration

```cuda
extern "C"
cudaError_t ternary_zero_gemv_f16(
    const uint32_t* __restrict__ weights,   // Packed 2-bit weights [M × (N/16)]
    const __half*   __restrict__ activations,// FP16 activations [N]
    __half*         __restrict__ output,     // FP16 output [M]
    int M, int N,
    cudaStream_t stream
);
```

**Grid/Block Dimensions:**
- Each block handles one output row (one row of W × full activation vector)
- Block size: 256 threads (8 warps)
- Each warp handles N/(16×256) iterations, processing 16 weights per thread per iteration
- Grid: `(M, 1, 1)` — embarrassingly parallel across output rows

### Step 2.5: Shared Memory & L2 Cache Strategy

**Shared Memory Usage:**
- Stage activation tile: `__shared__ __half s_act[TILE_N]` where `TILE_N = 1024`
- Each block loads a 1024-element activation tile cooperatively into shared memory
- All warps in the block read from smem during computation → 8× reduction in global mem reads
- smem usage: 1024 × 2 bytes = 2 KB per block (well within 48 KB limit)

**L2 Cache Policy (Persistent Tiles):**
```cuda
// Use cudaStreamSetAccessPolicy or cudaAccessPolicyWindow
// Pin the most frequently accessed weight rows in L2
cudaStreamAttrValue attr;
attr.accessPolicyWindow.base_ptr  = (void*)weights;
attr.accessPolicyWindow.num_bytes = l2_pin_size;  // e.g., 24 MB (full L2)
attr.accessPolicyWindow.hitRatio  = 1.0f;
attr.accessPolicyWindow.hitProp   = cudaAccessPropertyPersisting;
attr.accessPolicyWindow.missProp  = cudaAccessPropertyStreaming;
cudaStreamSetAttribute(stream, cudaStreamAttributeAccessPolicyWindow, &attr);
```

### Step 2.6: Warp-Level Reduction

```cuda
// After each warp accumulates its partial sum (half2 in registers):
// Step 1: __shfl_down_sync within warp (5 butterfly steps for warp=32)
half2 val = warp_accumulator;
for (int offset = 16; offset >= 1; offset >>= 1) {
    val += __shfl_down_sync(0xFFFFFFFF, val, offset);
}
// Step 2: First lane of each warp writes to shared memory
if (lane_id == 0) s_warp_sums[warp_id] = val;
__syncthreads();

// Step 3: First warp reduces across warps (8 warps → 3 iterations)
if (warp_id == 0) {
    half2 sum = (lane_id < num_warps) ? s_warp_sums[lane_id] : __float2half(0.0f);
    for (int offset = 4; offset >= 1; offset >>= 1)
        sum += __shfl_down_sync(0xFFFFFFFF, sum, offset);
    if (lane_id == 0) output[row] = sum;
}
```

### Step 3.2: `src/bitlinear.rs` — BitLinear Abstraction

```rust
use std::marker::PhantomData;
use half::f16;

pub struct GpuBuffer<T> {
    ptr: *mut T,
    len: usize,
}

impl<T> GpuBuffer<T> {
    pub fn alloc(len: usize) -> Result<Self, CudaError> { /* cudaMalloc */ }
    pub fn copy_from_host(&mut self, data: &[T]) -> Result<(), CudaError> { /* cudaMemcpy H2D */ }
    pub fn copy_to_host(&self, data: &mut [T]) -> Result<(), CudaError> { /* cudaMemcpy D2H */ }
}

impl<T> Drop for GpuBuffer<T> {
    fn drop(&mut self) { /* cudaFree */ }
}

/// Safety: GpuBuffer owns unique device memory
unsafe impl<T> Send for GpuBuffer<T> {}
unsafe impl<T> Sync for GpuBuffer<T> {}

pub struct BitLinear {
    packed_weights: GpuBuffer<u32>,   // [M × (N/16)]
    output_buffer: GpuBuffer<u16>,    // [M] — pre-allocated for kernel output
    stream: CudaStream,
    m: usize,
    n: usize,
    l2_pinned: bool,                  // tracks L2 cache policy state
}

impl BitLinear {
    pub fn new(m: usize, n: usize) -> Result<Self, CudaError> {
        assert!(n % 16 == 0, "N must be multiple of 16 (2-bit packing)");
        let packed_cols = n / 16;
        let packed_weights = GpuBuffer::alloc(m * packed_cols)?;
        let output_buffer = GpuBuffer::alloc(m)?;
        let stream = CudaStream::new()?;
        Ok(Self { packed_weights, output_buffer, stream, m, n, l2_pinned: false })
    }

    /// Pack ternary weights {-1, 0, 1} into 2-bit packed uint32_t format
    pub fn load_weights(&mut self, ternary_weights: &[i8]) -> Result<(), CudaError> {
        let packed = pack_ternary_to_u32(ternary_weights, self.n);
        self.packed_weights.copy_from_host(&packed)
    }

    /// Forward pass: y = W_ternary × x
    /// Note: bias is handled in the Python layer, not in the Rust/CUDA kernel
    pub fn forward(&mut self, activations: &[f16]) -> Result<Vec<f16>, CudaError> {
        assert_eq!(activations.len(), self.n);

        // Upload activations to GPU (allocated per-call; not pre-cached)
        let mut d_act: GpuBuffer<u16> = GpuBuffer::alloc(self.n)?;
        let act_u16: Vec<u16> = activations.iter().map(|h| h.to_bits()).collect();
        d_act.copy_from_host(&act_u16)?;

        // Launch kernel — writes into pre-allocated output_buffer
        unsafe {
            crate::ffi::ternary_zero_gemv_f16(
                self.packed_weights.as_ptr(),
                d_act.as_ptr(),
                self.output_buffer.as_mut_ptr(),
                self.m as i32,
                self.n as i32,
                self.stream.raw(),
            ).to_result()?;
            self.stream.synchronize()?;
        }

        // Read back results
        let mut output_u16 = vec![0u16; self.m];
        self.output_buffer.copy_to_host(&mut output_u16)?;
        let output: Vec<f16> = output_u16.iter().map(|&bits| f16::from_bits(bits)).collect();
        Ok(output)
    }
}
```

### Step 3.3: `src/ste.rs` — Straight-Through Estimator

```rust
/// Quantize FP16 weights to ternary {-1, 0, 1} using STE-aware rounding
/// During forward: hard quantization
/// During backward: gradient passes through (STE approximation)
pub fn ternary_quantize_ste(
    weights: &[f16],        // FP16 weights [M × N]
    alpha: f32,             // Threshold scaling factor (0.0 to 1.0)
) -> (Vec<i8>, f32) {
    // Compute mean absolute value
    let abs_sum: f32 = weights.iter().map(|w| w.to_f32().abs()).sum();
    let mean_abs = abs_sum / weights.len() as f32;
    let threshold = alpha * mean_abs;

    // Quantize and compute scale from non-zero weights only
    let mut scale_sum = 0.0f32;
    let mut scale_count = 0usize;
    let ternary: Vec<i8> = weights.iter().map(|&w| {
        let wf = w.to_f32();
        if wf.abs() > threshold {
            scale_sum += wf.abs();
            scale_count += 1;
            if wf > 0.0 { 1 } else { -1 }
        } else { 0 }
    }).collect();

    // Scale = mean of |w| for non-zero ternary weights
    // (NOT divided by total count — that would dilute the scale)
    let scale = if scale_count > 0 { scale_sum / scale_count as f32 } else { 1.0 };
    (ternary, scale)
}
```

> **Note:** The scale factor is the mean absolute value of weights that survived thresholding (non-zero ternary), not the mean of all weights. This ensures the dequantized approximation `w ≈ ternary × scale` minimizes reconstruction error for the non-zero entries.

## Phase 5: Performance Analysis — Compression Ratios & Memory Wall

### Compression Ratio Fundamentals

The compression ratio depends on the baseline precision. The mathematically accurate comparisons are:

| Baseline | Baseline Bits | Ternary Bits | Compression Ratio | Use Case |
|---|---|---|---|---|
| FP32 | 32 | 2 | **16×** | Standard training weights |
| FP16 / BF16 | 16 | 2 | **8×** | Inference-optimized weights |
| INT8 | 8 | 2 | **4×** | Quantized inference weights |

All subsequent analysis uses **FP16 as the primary baseline** (8×) since this kernel targets inference workloads where weights are already in half precision. When comparing against unoptimized FP32 training pipelines, the effective ratio is **16×**.

### Theoretical Throughput Analysis (RTX 4060)

| Metric | FP16 GEMV | Ternary-Zero GEMV | Improvement |
|---|---|---|---|
| Bits per weight | 16 | 2 | 8× vs FP16 (16× vs FP32) |
| Bytes per weight | 2.0 | 0.25 | 8× vs FP16 |
| Weights per 128-bit load | 8 | 64 | 8× more weights per transaction |
| Memory bandwidth utilization | 272 GB/s | 272 GB/s | Same bus — no hardware change |
| Effective weight throughput | 272 GB/s | ~2.17 TB/s (effective) | 8× theoretical ceiling |
| Arithmetic per byte loaded | 1 FMA | ~3 conditional ops | Branchless zero-gate |

### Detailed Breakdown

**1. Memory Bandwidth Dominance (and Its Limits):**
- GEMV is memory-bound: `bytes_loaded = M × N × sizeof(weight)`
- FP16: `M × N × 2 bytes`
- Ternary-Zero: `M × N × 0.25 bytes` (2-bit packed)
- **8× reduction** in weight memory traffic vs FP16

> **Memory Wall Caveat:** Weight compression addresses only one side of the bandwidth equation. The kernel must still stream activations (N × 2 bytes per row), write outputs (M × 2 bytes), and perform reduction operations. For small M (autoregressive decoding, M=1), activation streaming and output writes are negligible, and the 8× ratio is approachable. For larger M, the overhead of activation loads, shared memory synchronization, warp-level reductions, and host-device transfer progressively erode the theoretical gain. Speedup curves will flatten — expect **4-6× real-world** for typical dimensions, not the full 8×.

**2. L2 Cache Effectiveness:**
- RTX 4060 has 24 MB L2 cache
- A 4096×4096 weight matrix:
  - FP16: 32 MB (doesn't fit in L2)
  - Ternary: 4 MB (fits entirely in L2!)
- **Result:** After first touch, weights served from L2 at ~500 GB/s vs 272 GB/s from DRAM

> **Caveat:** L2 pinning (`cudaAccessPropertyPersisting`) is a best-effort hint, not a guarantee. The CUDA runtime may evict pinned tiles under memory pressure from other SMs. The effective L2 hit rate depends on the concurrent workload and should be profiled, not assumed.

**3. Zero-Gate Skip Benefit:**
- In sparse ternary models, ~30-50% of weights are typically zero
- Zero-gate skips accumulation for zero weights → reduces instruction count
- Additional ~1.3-1.8× effective speedup beyond the memory compression (varies with actual sparsity)

> **Caveat:** The zero-gate is branchless (mask-based), not a true branch skip. The GPU still executes the load and decode pipeline for zero weights; only the final `hadd2` accumulation is zeroed. The benefit comes from the masked accumulation producing a zero addend (no FP16 addition energy/latency), not from instruction elimination. True instruction skipping would require predicated execution at the warp level, which is not used here.

**4. Register Pressure & Occupancy:**
- 40 registers/thread target
- RTX 4060: 65536 registers/SM
- Max threads/SM at 40 regs: 65536/40 = 1638 → capped at 1536 (48 warps)
- At 256 threads/block: 6 blocks/SM → high occupancy
- **Result:** latency hiding through massive parallelism

**5. Realistic Throughput Estimate:**
- Theoretical DRAM bandwidth: 272 GB/s
- Weight traffic with 8× compression: 8× reduction → ~2.17 TB/s effective weight throughput
- **But the kernel also incurs:**
  - Activation tile loads: N × 2 bytes per output row (not compressed)
  - Shared memory sync: `__syncthreads()` per tile iteration
  - Warp reduction: 5 butterfly shuffle steps per warp + cross-warp reduction
  - Output writeback: M × 2 bytes
- For M=1, N=4096:
  - Weight bytes: 4096 × 0.25 = 1 KB
  - Activation bytes: 4096 × 2 = 8 KB (not compressed)
  - Output bytes: 1 × 2 = 2 bytes
  - **Total memory traffic ≈ 9 KB** (not just 1 KB)
  - Latency ≈ 9 KB / 272 GB/s ≈ 0.03 μs (compute is trivial; launch overhead dominates at this scale)
  - For larger N (e.g., N=4096, M=64): weight traffic = 64 KB, activation = 8 KB → weight-dominated, ~8× gain is realistic
- **Expect 4-6× end-to-end speedup over FP16 GEMV for production workloads, with diminishing returns as M grows (GEMV-to-GEMM transition).**

## Phase 6: Python Framework Layer

The Rust/CUDA kernel is wrapped by a full PyTorch-compatible Python framework (`ternary_zero/`). This layer is not a thin wrapper — it is a complete mini-framework with autograd, neural network modules, optimizers, and quantization utilities.

### 6.1: Tensor System (`ternary_zero/tensor.py`)
- `Tensor` class backed by `numpy.ndarray` with autograd metadata (`requires_grad`, `_grad_fn`, `grad`)
- Operator overloading (`__add__`, `__matmul__`, etc.) dispatches to autograd `Function` subclasses
- In-place operations (`fill_`, `zero_`, `normal_`) increment `_version` but no downstream checks read it
- `backward()` triggers reverse-mode autodiff via topological sort

### 6.2: Autograd Engine (`ternary_zero/autograd/`)
- `Function` base class with `apply()` pattern (similar to PyTorch's `torch.autograd.Function`)
- `engine.py`: reverse-mode AD via topological sort over `_grad_fn` back-pointers
- 20+ function implementations: `Add`, `Sub`, `Mul`, `Div`, `MatMul`, `ReLU`, `Softmax`, `CrossEntropyLoss`, `TernaryQuantizeSTE`, etc.
- **Key risk:** `_build_topo` uses recursion — deep graphs (100+ ops) will hit Python's recursion limit

### 6.3: Neural Network Modules (`ternary_zero/nn/`)
- `Module` base class with parameter registration, `train()`/`eval()` modes, `state_dict()` serialization
- `Linear`: standard dense layer with `Parameter` weights
- `BitLinear`: ternary-aware layer — during training uses STE quantization, during inference uses cached ternary weights
- Activations: `ReLU`, `GELU`, `Sigmoid`, `Tanh`, `Softmax`, `LogSoftmax`
- Normalization: `LayerNorm`, `BatchNorm1d`
- Losses: `CrossEntropyLoss`, `MSELoss`, `L1Loss`
- Containers: `Sequential`

### 6.4: Optimizers (`ternary_zero/optim.py`)
- `Optimizer` base class with param groups and state management
- `SGD` (with momentum, Nesterov), `Adam`, `AdamW`, `RMSprop`
- All operate on `numpy.ndarray` data — no GPU-accelerated optimizer steps

### 6.5: Quantization Utilities (`ternary_zero/quantize.py`)
- `ternary_quantize()`: STE-aware quantization (delegates to `TernaryQuantizeSTE` autograd function)
- `ternary_quantize_fixed()`: threshold-based quantization (no gradient)
- `pack_ternary_to_u32()` / `unpack_u32_to_ternary()`: 2-bit packing in pure Python/NumPy
- `ternary_weight_analysis()`: sparsity and compression statistics

### 6.6: Build & Distribution (`pyproject.toml`)
- Built with `maturin` (Rust → Python extension module)
- Produces `ternary_zero._core` native module with PyO3 bindings
- Falls back gracefully if CUDA is unavailable (`_HAS_NATIVE` flag in `__init__.py`)

## Risk Mitigation

### 1. PTX Portability — Architectural Fragility

The kernel relies on architecture-specific PTX inline assembly (`PRMT`, `LOP3.LUT`, `BFE`) in `kernel/ptx_utils.h`. These instructions are **not portable across GPU architectures**:

| Instruction | sm_80 (Ampere) | sm_89 (Ada Lovelace) | sm_90 (Hopper) | sm_86 (Ampere GA102) |
|---|---|---|---|---|
| `PRMT` | Yes | Yes | Yes | Yes |
| `LOP3.LUT` | Yes | Yes | Yes | Yes |
| `BFE` | Yes | Yes | Yes | Yes |

While these specific instructions are available on sm_80+, the register pressure assumptions (`-maxrregcount=64`), shared memory layout, and occupancy targets are tuned for sm_89 (RTX 4060). Running on different architectures may produce correct results but with suboptimal performance.

> **Current State (Research Prototype):** The PTX macros are used unconditionally with no fallback path. This is acceptable for a single-target research kernel but **not production-safe**.

> **Recommended Production Architecture:**
> ```
> kernel/
> ├── ternary_zero.cu              # Main kernel (dispatches at compile time)
> ├── ptx_utils.h                  # PTX intrinsics (sm_80+ only)
> ├── portable_utils.h             # Pure C++ bitwise fallback (any CUDA arch)
> └── arch_detect.h                # __CUDA_ARCH__ based dispatch macros
> ```
> Use `#if __CUDA_ARCH__ >= 800` guards to select PTX paths, with a portable C++ bitwise fallback (`uint32_t` masks and shifts) for older architectures. This ensures the kernel compiles and runs correctly on any CUDA-capable GPU, while still exploiting PTX on supported hardware.

### 2. FP16 Precision
- Accumulation in `half2` for vectorized ops; final warp reduction in `half`
- FP16 has ~3.3 decimal digits of precision; for N=4096, accumulated rounding error is bounded by `N × ε_fp16 ≈ 4096 × 0.001 ≈ 4.1` (absolute) — acceptable for inference, risky for training
- **Recommendation:** Final reduction should accumulate in `float` for production accuracy, converting to `half` only at output writeback

### 3. Build Complexity
- `build.rs` detects `CUDA_HOME` / `CUDA_PATH` env vars and finds `nvcc` automatically
- On Windows, uses `cc` crate to locate MSVC `cl.exe` / `lib.exe` for link path injection
- Compiles `kernel/ternary_zero.cu` → `.obj` → `.lib` (static library), linked with `cudart_static`

### 4. Memory Safety
- `GpuBuffer<T>` implements `Drop` for automatic `cudaFree`; `unsafe impl Send/Sync` only for device pointers
- RAII pattern prevents device memory leaks on panic or early return
- `CudaStream` RAII wrapper prevents stream handle leaks

### 5. Autograd Engine Reliability (Critical)

The Python autograd engine (`ternary_zero/autograd/`) is a non-trivial subsystem whose correctness is **as critical as kernel performance** for the project's utility. Known complexity areas:

- **Graph Lifecycle:** The computation graph is implicit (built via `_grad_fn` back-pointers). There is no explicit graph object, no cycle detection, and no reference counting. Deep graphs risk stack overflow in the topological sort (`_build_topo` uses recursion).
- **Gradient Accumulation:** Gradients are accumulated via `+=` into `.grad.data`. This is correct for standard use but interacts poorly with in-place operations — if a tensor is modified in-place after being saved by a `Function`, the backward pass reads stale data.
- **Broadcasting Semantics:** The `_unbroadcast` function in `engine.py` reduces gradients to match input shapes. Edge cases (scalar inputs, empty dimensions, multi-axis broadcast) require careful testing.
- **Memory Reuse:** No gradient checkpointing or graph pruning. Full forward activations are retained for backward pass, limiting model size on CPU-bound workloads.
- **Version Tracking:** `Tensor._version` is incremented on in-place ops but no `Function` checks it. This is a latent correctness bug — PyTorch detects version mismatches and raises errors; this engine does not.
